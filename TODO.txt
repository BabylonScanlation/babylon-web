Mirando el código y los logs de la consola, identifico los tres problemas claramente. Vamos uno por uno.

**Problema 3 (Anuncios):** El error `allow-scripts + allow-same-origin` hace que el iframe del ad no pueda navegar el top-frame, bloqueando redirects de los anuncios. Fix: agregar `allow-top-navigation-by-user-activation` al sandbox.

**Problema 2 (Imágenes dobles en Network):** Las imágenes se fetchean dos veces porque `_0xL` no marca el elemento como "en proceso" antes de que el async arranque, permitiendo doble-disparo.

**Problema 1 (Chunks):** Son chunks normales de Svelte/Vite, pero se pueden agrupar mejor.El archivo está en el contexto como documento, no como archivo. Te doy directamente los tres cambios exactos que debes aplicar en `src/layouts/Layout.astro`:

---

## Fix 2: Anuncios no muestran (error sandbox iframe)

Busca:

```js
ifr.setAttribute('sandbox', 'allow-scripts allow-same-origin allow-popups allow-forms');
```

Reemplaza por:

```js
ifr.setAttribute('sandbox', 'allow-scripts allow-same-origin allow-popups allow-forms allow-top-navigation-by-user-activation allow-pointer-lock');
```

El error `"allow-scripts and allow-same-origin can escape sandboxing"` es solo un *warning* de seguridad inevitable con Adsterra — no bloquea nada. El problema real era que el script del anuncio intentaba redirigir el top-frame pero sin `allow-top-navigation-by-user-activation` el navegador lo bloqueaba silenciosamente.

---

## Fix 3: Chunks excesivos (Vite)

En `astro.config.mjs`, dentro de `vite:`, agrega `build.rollupOptions`:

```js
vite: {
  ssr: {
    noExternal: ['firebase-admin'],
    external: ['@cloudflare/workers-types'],
  },
  optimizeDeps: {
    exclude: ['swiper', 'swiper/modules', '@cf-wasm/photon'],
  },
  build: {
    sourcemap: false,
    rollupOptions: {
      external: ['@cf-wasm/photon'],
      output: {
        // Agrupa todos los chunks de svelte internos en uno solo
        manualChunks(id) {
          if (id.includes('svelte')) return 'svelte-vendor';
          if (id.includes('node_modules')) return 'vendor';
        },
      },
    },
  },
},
```

## Resumen de causas

| Problema | Causa | Fix |
|---|---|---|
| Imágenes dobles en Network | `_0xL` sin guard de vuelo, el IntersectionObserver + la llamada directa disparaban dos fetches | Set `_0xP` como mutex |
| Anuncios bloqueados | Iframe sin `allow-top-navigation-by-user-activation` impedía que el ad hiciera redirect al hacer click | Agregar flag al sandbox |
| Chunks excesivos | Vite split automático de Svelte internals | `manualChunks` agrupa svelte en un solo vendor |

Las imágenes seguirán apareciendo en el Network tab (es imposible ocultarlas del DevTools), pero al abrir en nueva pestaña sin el header `X-Babylon-Service: nuclear-loader` ya devuelven **403**, que es la protección correcta.
