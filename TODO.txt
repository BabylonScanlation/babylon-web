## 1. CRÍTICO — Fallback secrets hardcodeados (CWE-798)

**Severidad:** Critical | **Archivos:** `src/actions/auth.ts`, `src/lib/session.ts`, `src/lib/nonce.ts`, `src/lib/crypto.ts`

```ts
// src/actions/auth.ts
const jwtSecret = env.JWT_SECRET || 'fallback-secret-change-me-in-production';

// src/lib/nonce.ts
const secret = runtime.env.JWT_SECRET || 'nonce-secret-fallback';

// src/lib/crypto.ts
const salt = import.meta.env.INTERNAL_CRYPTO_SALT || 'development-only-salt';
```

Si `JWT_SECRET` no está configurado en producción, **cualquier atacante puede firmar JWTs válidos** con la cadena pública `'fallback-secret-change-me-in-production'`. Esto permite escalada de privilegios total (`isAdmin: true`).

**Fix:**
```ts
const jwtSecret = env.JWT_SECRET;
if (!jwtSecret) throw new Error('JWT_SECRET env var is required');
```

## 2. CRÍTICO — Shield token hardcodeado en el cliente (CWE-798 / CWE-200)

**Severidad:** Critical | **Archivos:** `src/layouts/Layout.astro`, `src/pages/api/assets/proxy/[...key].ts`

El header `X-Shield-Token: 'babylon-v2-shield'` se compara en el proxy, pero el valor está expuesto en el JS del cliente (`Layout.astro`). Cualquier usuario puede leerlo en DevTools y hacer peticiones directas al proxy bypasseando la protección.

**Impacto encadenado con SSRF:** El proxy acepta keys que empiezan con `http` y las fetcha externamente:
```ts
// src/pages/api/assets/proxy/[...key].ts
if (objectKey.startsWith('http')) {
  const externalRes = await fetch(objectKey); // SSRF si el token es bypasseable
```

Con el token visible en cliente, un atacante podría usar el proxy para hacer fetch de URLs internas de Cloudflare o metadatos del worker.

**Fix mínimo:**
```ts
// Bloquear esquemas no-https y rangos IP privados antes del fetch externo
if (!objectKey.startsWith('https://')) {
  return new Response('Invalid scheme', { status: 400 });
}
// Validar host contra allowlist
const allowed = ['api.telegram.org', 'cdn.ejemplo.com'];
const url = new URL(objectKey);
if (!allowed.includes(url.hostname)) {
  return new Response('Forbidden host', { status: 403 });
}
```

## 3. ALTO — XOR "cifrado" trivialmente reversible (CWE-327)

**Severidad:** High | **Archivo:** `src/lib/obfuscator.ts`

```ts
const SALT = 'B4byl0n_V2_Nucl3ar_S3cur1ty_';
function xorTransform(str: string): string {
  const key = 42; // Llave fija
  return str.split('').map(c => String.fromCharCode(c.charCodeAt(0) ^ key)).join('');
}
```

El XOR con clave fija 42 y salt estático es **seguridad por oscuridad**, no cifrado. Los manifiestos de capítulos (URLs firmadas de imágenes) se envían al cliente obfuscados con esto. Cualquier atacante puede revertirlo en segundos:

```js
// PoC en consola del navegador
const SALT = 'B4byl0n_V2_Nucl3ar_S3cur1ty_';
const decode = s => atob(s.replace(/-/g,'+').replace(/_/g,'/'));
const xor = s => s.split('').map(c => String.fromCharCode(c.charCodeAt(0)^42)).join('');
const result = xor(decode(encryptedStr)).slice(SALT.length);
console.log(JSON.parse(result)); // URLs originales de R2
```

Las URLs ya van firmadas criptográficamente (`signManifest`), así que el obfuscador no añade seguridad real. Documentar esto como "ofuscación cosmética" y no depender de ella para controles de acceso.

## 4. ALTO — Rate limiter en memoria no funciona en Workers distribuidos (CWE-400)

**Severidad:** High | **Archivo:** `src/lib/middlewares/shield.ts`

```ts
const RATE_LIMIT = new Map<string, { count: number; resetTime: number }>();
```

Los Workers de Cloudflare corren en múltiples instancias/isolates. El `Map` se reinicia en cada cold start y **no se comparte entre instancias**. Un atacante con múltiples IPs puede enviar miles de peticiones.

**Fix:** Usar el KV binding ya disponible:
```ts
const record = await env.KV_VIEWS.get(`rl:${clientIp}`);
const count = record ? parseInt(record) + 1 : 1;
await env.KV_VIEWS.put(`rl:${clientIp}`, count.toString(), { expirationTtl: 60 });
if (count > MAX_REQ_PER_WINDOW) return new Response('Too Many Requests', { status: 429 });
```

## 5. ALTO — `clearSessionCache` es un stub no-op (CWE-613)

**Severidad:** High | **Archivos:** `src/lib/middlewares/auth.ts`, `src/actions/ui.ts`

```ts
export function clearSessionCache(_sessionId?: string) {
  // Legacy stub: session cache removed to prevent memory leaks in isolates
}
```

Se llama tras el toggle de NSFW y se supone que invalida el JWT del usuario. Pero es una función vacía. El JWT del usuario **sigue siendo válido** durante sus 5 minutos aunque se haya modificado `isNsfw` en DB. En un flujo de moderación esto podría importar.

**Fix:** Incrementar un campo `tokenVersion` en la tabla `Users` y verificarlo en el middleware:
```ts
// En authFlow: verificar tokenVersion del JWT contra DB si operación sensible
if (payload.tokenVersion !== dbUser.tokenVersion) {
  return deleteSession(context); // Forzar re-login
}
```

## 6. MEDIO — JWKS de Firebase sin caché (performance + disponibilidad)

**Severidad:** Medium | **Archivo:** `src/lib/firebase/server.ts`

```ts
const res = await fetch(JWKS_URL); // fetch en cada verificación de token
```

Cada login hace un fetch externo a `googleapis.com`. Si Google responde lento o hay un fallo de red, el login falla. Además consume cuota de requests de Workers.

**Fix:** Cachear en KV con TTL de 1 hora:
```ts
const cached = await env.KV_VIEWS.get('firebase_jwks');
if (cached) return JSON.parse(cached);
const fresh = await fetch(JWKS_URL).then(r => r.json());
await env.KV_VIEWS.put('firebase_jwks', JSON.stringify(fresh), { expirationTtl: 3600 });
```

## 8. MEDIO — `Function constructor` como anti-debug (CWE-95 semántico)

**Severidad:** Medium | **Archivo:** `src/layouts/Layout.astro`

```js
setInterval(() => { (function(){return false;}['constructor']('debugger')['call']()); }, 200);
```

Esta técnica usa `Function constructor` (equivalente a `eval`). Algunos WAFs/CSP la bloquean. Si tienes un CSP header que restringe `unsafe-eval`, este código falla silenciosamente. Más importante: **no detiene a ningún atacante real** — es trivial bypassearlo con `Object.defineProperty(window, 'debugger', {})` o simplemente con Puppeteer.

Recomendar eliminarlo o reemplazarlo por protecciones más robustas a nivel de infraestructura.
